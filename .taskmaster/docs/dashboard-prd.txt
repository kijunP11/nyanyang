# Overview  
나냥 웹사이트의 대시보드 페이지를 Figma 디자인에 맞춰 구현합니다. 현재 템플릿 상태인 대시보드를 실제 콘텐츠를 표시하는 기능적인 페이지로 전환합니다. 사용자가 로그인 후 접속하는 메인 대시보드로, 웹소설/콘텐츠 플랫폼의 특성에 맞는 콘텐츠 카드 그리드 레이아웃을 제공합니다. 

# Core Features  

## 1. 섹션 헤더 컴포넌트
- 섹션 제목 표시 (예: "Safe & Fun 공모전 수상작!")
- 재사용 가능한 컴포넌트로 다양한 섹션 타입 지원 (공모전, 인기작, 신작 등)
- 각 섹션은 독립적으로 관리 가능

## 2. 콘텐츠 카드 그리드 레이아웃
- 반응형 그리드 레이아웃 구현 (모바일: 1열, 태블릿: 2열, 데스크톱: 3-4열)
- 각 카드는 305px × 138px 크기 (Figma 디자인 기준)
- 카드 간 적절한 간격 및 정렬 유지

## 3. 콘텐츠 카드 컴포넌트
각 카드는 다음 요소들을 포함:
- 썸네일 이미지: 왼쪽에 위치 (89px × 138px), lazy loading 지원
- 제목: 카드 상단에 표시 (24px 높이), 긴 제목은 truncate 처리
- 통계 정보 행: 조회수, 좋아요, 댓글 수를 아이콘과 함께 표시
  - 조회수 아이콘 (Eye) + 숫자
  - 좋아요 아이콘 (Heart) + 숫자 (예: 3.7k, 17k 형식)
  - 댓글 수 아이콘 (MessageCircle) + 숫자
- 설명 텍스트: 제목 아래에 표시 (16px 또는 32px 높이, 2줄 제한)
- 작성자 정보: 하단에 @username 형식으로 표시, 클릭 시 프로필 이동
- 태그/배지: 하단에 18px × 18px 크기의 배지들 (여러 개 가능)

## 4. 데이터 포맷팅 및 표시
- 숫자 포맷팅: 1000 이상은 "k" 단위로 표시 (예: 3700 → "3.7k", 17000 → "17k")
- 텍스트 truncate: 긴 제목과 설명은 말줄임표 처리
- 이미지 로딩 및 fallback 처리

## 5. 인터랙션 및 사용자 경험
- 카드 호버 효과 및 트랜지션 애니메이션
- 카드 클릭 시 상세 페이지로 이동 (React Router Link)
- 작성자 이름 클릭 시 작성자 프로필로 이동
- 로딩 상태: 스켈레톤 UI 표시
- 빈 상태: 콘텐츠가 없을 때 적절한 메시지 표시

# User Experience  

## User Personas
- 일반 사용자: 콘텐츠를 탐색하고 읽고 싶어하는 사용자
- 작성자: 자신의 작품이 어떻게 표시되는지 확인하고 싶은 사용자

## Key User Flows
1. 콘텐츠 탐색 플로우: 로그인 → 대시보드 접속 → 섹션별 콘텐츠 카드 확인 → 관심 있는 카드 클릭 → 상세 페이지 이동
2. 작성자 프로필 확인 플로우: 카드의 작성자 이름 클릭 → 작성자 프로필 페이지로 이동

## UI/UX Considerations
- 반응형 디자인: 모바일, 태블릿, 데스크톱 모두 지원
- 접근성: 키보드 네비게이션 및 스크린 리더 지원
- 성능: 이미지 lazy loading, 불필요한 리렌더링 방지

# Technical Architecture  

## System Components
- 컴포넌트 구조:
  - `app/features/users/components/content-card.tsx` - 콘텐츠 카드 컴포넌트
  - `app/features/users/components/section-header.tsx` - 섹션 헤더 컴포넌트
  - `app/features/users/components/stats-row.tsx` - 통계 정보 행 컴포넌트
  - `app/features/users/components/content-grid.tsx` - 그리드 레이아웃 컴포넌트
  - `app/features/users/screens/dashboard.tsx` - 대시보드 메인 페이지
  - `app/features/users/types/content.ts` - 콘텐츠 관련 타입 정의
  - `app/features/users/lib/formatters.ts` - 숫자/텍스트 포맷팅 유틸리티

## Data Models
```typescript
interface ContentItem {
  id: string;
  title: string;
  description: string;
  thumbnailUrl: string;
  author: {
    id: string;
    username: string;
    avatarUrl?: string;
  };
  stats: {
    views: number;
    likes: number;
    comments: number;
  };
  tags?: string[];
  createdAt: Date;
  updatedAt: Date;
}

interface Section {
  id: string;
  title: string;
  type: 'contest' | 'popular' | 'new' | 'custom';
  items: ContentItem[];
}
```

## APIs and Integrations
- React Router의 `loader` 함수를 사용한 서버 사이드 데이터 로딩
- Supabase를 통한 데이터베이스 쿼리 (현재는 mock 데이터로 시작)
- 기존 shadcn/ui 컴포넌트 활용: Card, Avatar, Badge
- lucide-react 아이콘: Eye, Heart, MessageCircle

## Infrastructure Requirements
- 기존 React Router + Supabase 인프라 활용
- Tailwind CSS를 사용한 스타일링
- 반응형 브레이크포인트: 모바일 < 768px (1열), 태블릿 768px-1024px (2열), 데스크톱 > 1024px (3-4열)

# Development Roadmap  

## Phase 1: MVP - 기본 구조 및 타입 정의
**목표**: 개발을 위한 기반 구조 구축

1. 타입 정의 파일 생성 (`app/features/users/types/content.ts`)
   - `ContentItem` 인터페이스 정의
   - `Section` 인터페이스 정의
   - 관련 유틸리티 타입 정의

2. Mock 데이터 생성
   - Figma 디자인 기반 샘플 데이터 생성
   - 다양한 통계 수치 포함 (k 단위 포함: 3.7k, 17k 등)
   - 다양한 설명 텍스트 길이 포함

3. 유틸리티 함수 생성 (`app/features/users/lib/formatters.ts`)
   - 숫자 포맷팅 함수: `formatNumber(3700) → "3.7k"`
   - 텍스트 truncate 함수: `truncateText(text, maxLength)`

## Phase 2: 컴포넌트 개발 - 개별 컴포넌트 구현
**목표**: 재사용 가능한 컴포넌트들 구현

1. `StatsRow` 컴포넌트 (`app/features/users/components/stats-row.tsx`)
   - 아이콘 + 숫자 표시
   - lucide-react 아이콘 활용 (Eye, Heart, MessageCircle)
   - 숫자 포맷팅 함수 적용

2. `ContentCard` 컴포넌트 (`app/features/users/components/content-card.tsx`)
   - 썸네일 이미지 영역 (왼쪽, 89px × 138px)
   - 콘텐츠 영역 (오른쪽, 200px 너비)
     - 제목 (24px 높이)
     - StatsRow 통합
     - 설명 텍스트 (16px 또는 32px 높이, 2줄 제한)
     - 작성자 정보 (@username 형식)
     - 태그/배지 (18px × 18px, 여러 개 가능)
   - 호버 효과 및 트랜지션
   - React Router Link로 상세 페이지 이동

3. `SectionHeader` 컴포넌트 (`app/features/users/components/section-header.tsx`)
   - 섹션 제목 표시
   - 재사용 가능한 구조

## Phase 3: 레이아웃 및 통합
**목표**: 컴포넌트들을 조합하여 완성된 레이아웃 구현

1. `ContentGrid` 컴포넌트 (`app/features/users/components/content-grid.tsx`)
   - 반응형 그리드 레이아웃 구현
   - ContentCard들을 그리드로 배치
   - 적절한 간격 및 정렬 (카드 크기: 305px × 138px)
   - 모바일: 1열, 태블릿: 2열, 데스크톱: 3-4열

2. `dashboard.tsx` 수정 (`app/features/users/screens/dashboard.tsx`)
   - SectionHeader 통합
   - ContentGrid 통합
   - Mock 데이터 연결
   - 기본 레이아웃 완성

## Phase 4: 스타일링 및 반응형 처리
**목표**: Figma 디자인에 맞춘 정확한 스타일링

1. 디자인 시스템 적용
   - Figma 디자인의 색상 팔레트 적용
   - 폰트 크기 및 가중치 적용
   - 간격 및 패딩 정확히 재현

2. 반응형 디자인
   - 모바일 레이아웃 (1열) 구현 및 테스트
   - 태블릿 레이아웃 (2열) 구현 및 테스트
   - 데스크톱 레이아웃 (3-4열) 구현 및 테스트
   - 브레이크포인트별 검증

3. 인터랙션 효과
   - 카드 호버 효과 구현
   - 트랜지션 애니메이션 추가
   - 포커스 상태 스타일링

## Phase 5: 데이터 연동 준비
**목표**: 실제 데이터 연동을 위한 준비

1. Loader 함수 구현
   - `dashboard.tsx`에 loader 함수 추가
   - Mock 데이터 반환 (초기)
   - 향후 Supabase 쿼리로 대체 가능한 구조 설계

2. 에러 처리 및 빈 상태
   - 데이터 로딩 실패 시 에러 처리
   - 빈 상태 UI 구현 (콘텐츠가 없을 때)

3. 로딩 상태
   - 스켈레톤 UI 구현
   - 로딩 중 표시

## Phase 6: 최적화 및 개선
**목표**: 성능 및 사용자 경험 개선

1. 이미지 최적화
   - Lazy loading 구현
   - 이미지 fallback 처리
   - 적절한 이미지 크기 설정

2. 성능 최적화
   - 불필요한 리렌더링 방지 (React.memo, useMemo)
   - 메모이제이션 적용
   - 가상화 고려 (대량 데이터 시)

3. 접근성 개선
   - 키보드 네비게이션 지원
   - ARIA 레이블 추가
   - 스크린 리더 지원

# Logical Dependency Chain

## Foundation Layer (Phase 1)
- 타입 정의 및 Mock 데이터가 모든 후속 작업의 기반이 됨
- 유틸리티 함수는 컴포넌트에서 즉시 사용됨
- **우선순위: 최우선**

## Component Layer (Phase 2)
- `StatsRow` → `ContentCard`에서 사용됨
- `ContentCard` → `ContentGrid`에서 사용됨
- 각 컴포넌트는 독립적으로 개발 및 테스트 가능
- **우선순위: Phase 1 완료 후**

## Integration Layer (Phase 3)
- 개별 컴포넌트들이 완성된 후 통합
- `dashboard.tsx`는 모든 컴포넌트를 조합
- **우선순위: Phase 2 완료 후**

## Polish Layer (Phase 4-6)
- 기본 기능 완성 후 스타일링 및 최적화
- 사용자 경험 개선은 기능 완성 후 진행
- **우선순위: Phase 3 완료 후**

## Development Order:
1. 타입 정의 (Phase 1.1) - 가장 먼저
2. Mock 데이터 (Phase 1.2) - 타입 정의 후
3. 유틸리티 함수 (Phase 1.3) - 타입 정의 후
4. StatsRow 컴포넌트 (Phase 2.1) - 유틸리티 함수 후
5. ContentCard 컴포넌트 (Phase 2.2) - StatsRow 후
6. SectionHeader 컴포넌트 (Phase 2.3) - 독립적, 병렬 가능
7. ContentGrid 컴포넌트 (Phase 3.1) - ContentCard 후
8. dashboard.tsx 통합 (Phase 3.2) - 모든 컴포넌트 후
9. 스타일링 (Phase 4) - 통합 후
10. 데이터 연동 (Phase 5) - 스타일링 후
11. 최적화 (Phase 6) - 모든 기능 완성 후

# Risks and Mitigations  

## Technical Challenges
1. Figma 디자인의 정확한 재현
   - 위험: 디자인의 세부 사항(간격, 색상, 폰트)을 정확히 재현하기 어려울 수 있음
   - 완화: Figma에서 정확한 CSS 값 추출, 디자인 시스템 문서화, 단계별 스타일링 및 검증

2. 반응형 디자인 구현
   - 위험: 다양한 화면 크기에서 레이아웃이 깨질 수 있음
   - 완화: 모바일 퍼스트 접근, 각 브레이크포인트별 테스트, Tailwind의 반응형 유틸리티 활용

3. 이미지 로딩 및 최적화
   - 위험: 많은 이미지로 인한 성능 저하
   - 완화: Lazy loading 구현, 이미지 최적화 (WebP 형식), CDN 활용

## MVP 범위 결정
- MVP에 포함: 기본 카드 레이아웃, Mock 데이터로 동작, 반응형 그리드 (모바일/데스크톱), 기본 스타일링
- MVP에 제외 (향후 개선): 실제 데이터베이스 연동, 무한 스크롤, 필터링/정렬 기능, 검색 기능

## Resource Constraints
- 시간 제약: Phase별로 나누어 점진적 개발, 각 Phase 완성 후 검토 및 조정, MVP 먼저 완성 후 개선
- 디자인 변경 가능성: 재사용 가능한 컴포넌트 구조, 유연한 레이아웃 시스템, 컴포넌트별 독립적 수정 가능
